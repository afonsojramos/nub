Nub's default install layout links every package into a **machine-global store**: one copy of each package version per machine, shared by every project that depends on it. Inside a project, `node_modules` is two levels of symlinks deep:

```console
$ readlink node_modules/ms
.nub/ms@2.1.3/node_modules/ms

$ readlink node_modules/.nub/ms@2.1.3
/Users/you/.cache/nub/pm/virtual-store/ms@2.1.3-b96c27c5c15521b2
```

This is pnpm's isolated layout with the virtual store moved one level up: pnpm keeps its store inside each project (`node_modules/.pnpm`) and hardlinks every file into it on every install, while Nub keeps the store in the machine cache — so once a package version has materialized anywhere, installing it into the next project is a symlink, not thousands of file operations. These are warm-reinstall numbers — a populated store, `node_modules` cleared — which is exactly the case the global store exists to make cheap. A cold install is a different workload.

Making this layout the *default* is the hard part. A cross-project store means a package's real path is outside the project — and a surprising amount of the ecosystem quietly assumes it isn't. This post is about the three mechanisms that close that gap.

## The catch: real paths

Node resolves modules through symlinks. When a store-resident package imports something, Node walks `node_modules` directories upward from the file's *real* location — and that walk climbs the machine cache, never the project:

```text
~/.cache/nub/pm/virtual-store/ms@2.1.3-…/node_modules/ms/index.js
~/.cache/nub/pm/virtual-store/ms@2.1.3-…/node_modules/   ← the deps ms declares
~/.cache/nub/pm/…                                        ← nothing of yours up here
```

For declared dependencies this is fine — every package's declared deps are linked next to it in the store, which is the whole point of an isolated layout. What breaks is everything that assumed the walk would pass through the project: undeclared imports, dev servers that gate file access by path prefix, and bundlers that scope themselves to the project directory. One mechanism for each.

## Phantom dependencies

A phantom dependency is an import of a package the importer never declared. Real packages ship them — `@inkjs/ui` imports `react` without declaring it; `@crawlee/basic` requires `@apify/datastructures`. Under a flat npm-style layout these resolve by accident. Under pnpm's project-local store, a hidden fallback tree inside the project catches many of them. A machine-global store has no project to fall back to: the importer's walk climbs the cache and the import throws.

Nub's answer is detection rather than curation. During install, as each tarball is imported into the store, a scanner parses that version's published code — walking the module graph from its entry points and checking every static, unguarded import against the dependencies it declares. Package contents are immutable, so the verdict is cached per content fingerprint: each version is scanned once per machine, and the scan runs during the network-bound fetch phase, where the CPU is otherwise idle.

At link time, a flagged package is **ejected**: materialized into the project as real files instead of a store symlink, so its real path — and its resolution walk — is back inside the project. Two details make the eject sound:

- **The ancestor closure comes along.** Ejecting only the offender would leave its store-resident importers resolving the shared copy — two real paths, two module instances, a silent singleton split. So every package that transitively imports the offender materializes with it. The closure is small in practice: 0.3–2.1% of the tree, measured on real large dependency graphs.
- **Unreachable targets get hoisted within.** When the undeclared target still can't be reached from the ejected copy, it is linked as a sibling inside the ejected package's own `node_modules`.

There is no hand-maintained package list to go stale. The verdicts come from the published code of the exact versions in your lockfile, and a scanner improvement re-scans cached content rather than serving old verdicts. ([#302](https://github.com/nubjs/nub/pull/302), [#319](https://github.com/nubjs/nub/pull/319), [#321](https://github.com/nubjs/nub/pull/321), [#328](https://github.com/nubjs/nub/pull/328))

The same treatment covers packages that write into their own install directory. Prisma's `generate` step writes the generated client next to `@prisma/client` on disk — under a shared store that location is machine-global and keyed by version, so two projects with different schemas would silently overwrite each other's client. Packages like that materialize per-project, and the store copy stays pristine.

## Serving from the store: Vite

Vite's dev server refuses to serve files from outside its allow-list: every `/@fs` request is checked against `server.fs.allow`, which defaults to the workspace root. A store-resident module served raw — a framework client entry, an SSR dep — fails with `403 … outside of Vite serving allow list` — something pnpm never hits, because its store sits under the workspace root.

The fix is one file. After every install with Vite in the graph, Nub writes the store's location where Vite can find it:

```console
$ cat node_modules/.modules.yaml
{"virtualStoreDir":"/Users/you/.cache/nub/pm/virtual-store"}
```

Vite 8.1+ reads this natively ([vitejs/vite#22415](https://github.com/vitejs/vite/pull/22415)) and appends the store directory to its allow-list. For older Vite — still most of what's installed — Nub backports the same behavior: it materializes the project's own Vite copy and inserts the equivalent check into the bundled dist, at Vite's own allow-list computation site, appending the store dir to whatever `fs.allow` the app or framework configured. The shared store copy is never patched — a guard refuses to modify anything under the store — and if the insertion point isn't found, nothing is patched and Vite runs unmodified. ([#318](https://github.com/nubjs/nub/pull/318), [#346](https://github.com/nubjs/nub/pull/346))

Both halves live on disk in `node_modules`, not in Nub's process. A teammate running plain `vite dev` against the same tree, or a CI job that never invokes Nub, gets the same working behavior — the compat mechanism has zero runtime coupling to Nub itself.

## Bundlers that refuse out-of-project paths

Some resolvers never consult Node's walk at all. Turbopack — Next.js's bundler — canonicalizes paths through symlinks and scopes file access to a single project root. Metro, as configured by bare React Native, crawls the project directory by real path, so with a machine-global store it fails to resolve even *declared* dependencies. These aren't phantom-dependency bugs; the imports are declared and correct. The resolver simply refuses files outside the project, by design.

No per-package ejection fixes that — the store itself has to be inside the project. So Nub keeps a short exclusion list: a project that depends on one of these tools automatically gets a **project-local store** instead. Same isolated layout, same phantom protection; the store lives under `node_modules/.nub` rather than the machine cache, at the cost of cross-project sharing for that project. Today the list has two entries — Next.js and React Native (Expo's Metro config is store-aware and works either way). It extends in `.npmrc`:

```ini
disableGlobalVirtualStoreForPackages[]=my-bundler
```

The list is a last resort, and it shrinks under root-causing. Parcel used to be on it — until the actual break turned out to be a Nub bug, not a Parcel one: the store prewarm hashed a differently-filtered dependency graph than the link phase, so `@parcel/core` materialized at two store paths, two copies loaded into one process, and Parcel's worker farm failed at startup with a `DataCloneError`. Fixing the hash mismatch removed the entry, and Parcel now builds under the global store. ([#338](https://github.com/nubjs/nub/pull/338))

## Where the global store doesn't apply

In CI, and under `nub ci`, every install is project-local regardless — real directories and relative links, a `node_modules` that survives a Docker `COPY --from` into a fresh image where the machine store wouldn't exist. And the layout stays a choice, per project or per install:

```ini
# keep isolation, move the store inside the project
enableGlobalVirtualStore=false

# flat, npm-style node_modules — no virtual store at all
node-linker=hoisted
```

The default, though, is the global store — with the scanner, the Vite compat, and the exclusion list closing the gaps that would otherwise make a cross-project store a leaky abstraction. See [the virtual store](/docs/install/virtual-store) for the layout options in full, and [store and disk layout](/docs/install#store-and-disk-layout) for how the pieces fit.
