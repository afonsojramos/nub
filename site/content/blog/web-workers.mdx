JavaScript has had a standard API for running code on another thread since 2009: the web's `Worker`.

It is the shape everyone writes. You construct a `Worker` pointing at a file, the file runs on its own thread with its own event loop, and the two sides talk through `postMessage`. This exact pair of files runs unchanged in every major browser, in Deno, and in Bun:

```ts
// main.js
const worker = new Worker(new URL("./worker.js", import.meta.url), { type: "module" });

worker.postMessage({ n: 41 });
worker.onmessage = (ev) => {
  console.log(ev.data); // ‚Üí 42
  worker.terminate();
};
```

```ts
// worker.js
self.onmessage = (ev) => self.postMessage(ev.data.n + 1);
```

Node is the exception:

```console
$ node main.js
const worker = new Worker(new URL("./worker.js", import.meta.url), { type: "module" });
               ^
ReferenceError: Worker is not defined

Node.js v26.2.0
```

## The most-upvoted issue on Node's tracker

Node is not missing threads. It has had real ones since 2018, behind `node:worker_threads` ‚Äî a capable, low-level primitive with its own `Worker` class, a `parentPort`, and structured-clone message passing. What it never shipped is the *web's* `Worker`: a different class, a different constructor, a different event surface, and no global. Code written for the web platform ‚Äî a library that offloads parsing to a worker, WASM glue that spawns a thread behind the scenes ‚Äî reaches for a global that is not there, and gets ported by hand instead.

Browser-standard `Worker` support is the most-upvoted open issue on the Node repository ‚Äî [nodejs/node#43583](https://github.com/nodejs/node/issues/43583), open since 2022, ahead of HTTP/3. It has not shipped.

## Why Node declined

The Node team's position is reasonable. A worker's *child* side ‚Äî the `DedicatedWorkerGlobalScope` a browser provides, with `importScripts`, `WorkerNavigator`, `WorkerLocation`, and an origin model ‚Äî is a different environment from a Node thread, and a global that looked browser-shaped on the parent while running a Node environment in the child would promise more than the runtime can deliver. The maintainers have been explicit that an incomplete polyfill is, if anything, an argument against pulling the feature into core.

For a standard library, that is a sound call ‚Äî a half-match is worse than a clear "use `node:worker_threads`." For a tool whose job is to bridge the web platform onto the Node you already run, the calculus is different: make no claim to be a browser, document exactly where the wrapper diverges, and give people the shape they already write. So Nub ships it.

## The same two files, on stock Node

Here is the pair from the top, unchanged, on the `node` your project already pins ‚Äî down to Node 18.19:

```console
$ nub main.js
42
```

And because Nub transpiles every entry it loads, the worker file can be TypeScript ‚Äî point the constructor at a `.ts` file and go, no build step.

## A polyfill, not a runtime

Nub's `Worker` rides `node:worker_threads` ‚Äî not a fork of Node, not a new runtime, not a patched binary. A preload defines `globalThis.Worker` as an `EventTarget` subclass wrapping a `node:worker_threads.Worker`, and inside the thread installs the dedicated-worker scope ‚Äî `self`, `postMessage`, `close`, `importScripts` ‚Äî on top of `parentPort`.

- **Feature-detected.** It installs only when no native global is present ‚Äî the day Node ships its own `Worker`, Nub steps aside.
- **Lazy.** Nothing loads until the first `new Worker(...)`, so a program that never spawns a worker pays nothing at startup.
- **Conformance-tested.** The surface runs a vendored slice of [web-platform-tests](https://github.com/web-platform-tests/wpt) ‚Äî the `webmessaging` battery, structured clone over a `MessageChannel`, and the worker-scope event tests ‚Äî on every supported Node tier in CI. Messaging and structured clone pass in full; the documented exceptions are browser-DOM types Node has no substrate for, like `ImageBitmap` and `OffscreenCanvas`.

## Using it

Everything below runs today under Nub, on stock Node.

**Module or classic.** The default shape is `{ type: "module" }`; pass `{ type: "classic" }` and the worker gets a synchronous `importScripts()`.

```ts
const classic = new Worker(new URL("./legacy.js", import.meta.url), { type: "classic" });
// inside legacy.js: importScripts("./helpers.js") ‚Äî fetched and run synchronously, in order
```

**An inline worker.** A `data:` URL runs a worker straight from a source string; a `blob:` URL from `URL.createObjectURL` is the WHATWG inline-worker mechanism, snapshotted at `createObjectURL` time.

```ts
const src = `self.onmessage = (e) => self.postMessage(e.data * 2);`;

const data = new Worker(`data:text/javascript,${encodeURIComponent(src)}`);

const blob = new Worker(
  URL.createObjectURL(new Blob([src], { type: "text/javascript" })),
);
```

**Channels.** Hand a worker one end of a `MessageChannel` and talk over it directly ‚Äî the ports transfer across the thread boundary.

```ts
const { port1, port2 } = new MessageChannel();

const worker = new Worker(new URL("./worker.ts", import.meta.url), { type: "module" });
worker.postMessage({ port: port2 }, [port2]);

port1.onmessage = (ev) => console.log(ev.data);
port1.postMessage("over the side channel");
```

**Broadcast.** A `BroadcastChannel` fans a message out to every thread tuned to the same channel name ‚Äî a native Node global, so it works unchanged.

```ts
const channel = new BroadcastChannel("jobs");
channel.onmessage = (ev) => console.log("job:", ev.data);
channel.postMessage({ id: 7, kind: "resize" });
```

**Transferables.** Move an `ArrayBuffer` instead of copying it ‚Äî it detaches on the sending side, zero-copy.

```ts
const buffer = new ArrayBuffer(1 << 20); // 1 MiB

worker.postMessage(buffer, [buffer]);
console.log(buffer.byteLength); // 0 ‚Äî transferred, not cloned
```

**Node-style code, on the same handle.** The global also mirrors `node:worker_threads.Worker` ‚Äî the `EventEmitter` methods, the `online` and `exit` lifecycle events, a `Promise`-returning `terminate()`, and `{ eval: true }` for a raw source string ‚Äî so web-style and Node-style worker code interoperate on one object.

```ts
const worker = new Worker(
  `const { parentPort } = require("node:worker_threads");
   parentPort.on("message", (n) => parentPort.postMessage(n * 2));`,
  { eval: true },
);

worker.on("message", (n) => console.log(n)); // 42 ‚Äî the raw value, Node-style
worker.postMessage(21);
```

<Callout title="One worker is one OS thread">
Each `Worker` is a real OS thread with its own V8 isolate and a fresh module graph ‚Äî heavier to start than a browser worker. Use a small pool for hot paths; don't spawn one per task. The full surface, the transferable set, and the documented divergences from a browser `Worker` are on the [Web Workers reference](/docs/runtime/workers).
</Callout>

## No new API to learn, nothing to undo

There is no Nub-specific `Worker`. It is the WHATWG Web Workers shape ‚Äî the same constructor, the same `postMessage`, the same events you already write for the browser ‚Äî running on the Node you already have. Remove Nub tomorrow and that code is still correct web-platform code; all you lose is the global.
