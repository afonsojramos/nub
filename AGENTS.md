This file is the entry point for AI coding agents working in this repository. It mirrors the [`AGENTS.md` convention](https://agents.md) used by other AI tools. [`CLAUDE.md`](CLAUDE.md) is a symlink to this file — Claude Code and the `AGENTS.md` convention read identical content.

It is an optional overlay; nothing in this file depends on it, and a contributor working without it has everything needed here.

## Non-negotiables — read first if you're opening a PR or editing this repo

A fast-read index of the load-bearing rules a doing-agent must not miss. Each links to its authoritative section below — read that section before acting on the area it governs. This block is an index, NOT a re-explanation; the full rules are canonical.

- **A PR that resolves an issue MUST have `Closes #N` / `Fixes #N` / `Resolves #N` in its BODY — NON-NEGOTIABLE.** Verify it's there before `gh pr create`. (See [Git & GitHub maintainer hygiene](#git--github-maintainer-hygiene-high-priority).)
- **The repo is PUBLIC — never commit internal/competitive discussion** (strategy, benchmark-framing, private attributions) to any tracked file or commit message. (See [The repo is PUBLIC](#the-repo-is-public--never-commit-internal-discussion-high-priority).)
- **Brand boundary (public surfaces only):** no public `globalThis.nub`, no `nub:*` import namespace, no `@nub/*` scope, no `"nub"` user-authored config field, no documented `NUB_*` user knob. Internals are exempt. (See [The brand boundary](#the-brand-boundary--public-surfaces-only-internals-are-exempt).)
- **`vendor/aube` changes follow fork-discipline:** default-preserving (standalone aube byte-for-byte unchanged on the default path; the embedder opts in) OR a justified latent-bug-fix jdx would accept. (See [The aube vendoring + upstreaming](#the-aube-vendoring--upstreaming--memorize-this-forgotten-twice-ground-every-aube-claim-here-not-memory).)

## Core design positions — ALWAYS HOLD THESE IN CONTEXT (read before any CLI/PM/runtime change)

These are the load-bearing, always-true architectural premises of nub. They are easy to violate by reflex (e.g. adding an npm flag, augmenting hijacked `node`), so keep them resident — do NOT design or dispatch against this surface without them. When a change touches the CLI grammar, the PM, or the runtime, FIRST re-state the relevant position here and design to it.

- **What nub is:** a **Rust CLI that AUGMENTS the user's installed Node** via Node's own extension surfaces (`--import`/`--require` preload, `module.registerHooks`, env, N-API addons, V8-flag injection). It is **NOT a fork**, ships no patched Node, embeds no libnode. The test for any augmentation: "would a user on plain Node + the corresponding `module.register()`/`--import`/npm-addon get the same result?"
- **The PM CLI frontend is pnpm-compatible — ONLY pnpm.** The package-manager command grammar mirrors **pnpm's, exclusively** — NOT npm, NOT yarn, NOT bun. When adding/checking a flag/alias/positional on a PM verb, the ONLY question is *"does real pnpm accept it?"* — never "does npm?". Do not add npm-isms (`--omit`, `--no-save`, `-S`/`--save`, npm's `-w <name>`/`--workspaces` selectors). Gotcha: pnpm `-w` = `--workspace-root` (boolean); member selection is `--filter`, recursion is `-r`. (This premise was violated once by grounding a compat audit against npm+pnpm — never again.)
  - **The DELIBERATE exceptions to pnpm-compat (decided 2026-06-20):** (1) the **top-level file-run command** (`nub <file>`) is nub's own surface, not pnpm's; (2) **`nub upgrade`** is nub's SELF-UPDATE (not pnpm's `upgrade`→`update` alias) — keep as-is, `nub upgrade <pkg>` does not route to `update`; (3) **NO implicit script shortcuts, EVER** — nub ALWAYS requires explicit `nub run <script>`; it never accepts `nub test`/`nub start`/`nub t`/etc. as run-shortcuts even though pnpm does; (4) **`nub init`** is nub's OWN TypeScript-first project scaffold (decided 2026-07-21, [#487](https://github.com/nubjs/nub/issues/487)) — NOT pnpm's bare non-interactive manifest write; it takes no positionals (like pnpm) and hints `nubx create-<template>` for templates (design record: `wiki/commands/init.md`, impl `crates/nub-cli/src/init.rs`). These four are the known, intentional divergences; everything else mirrors pnpm.
  - **The `nub pm <verb>` sub-namespace is nub's OWN PM-management surface, NOT a pnpm mirror.** `nub pm which`/`pin`/`migrate`/`update`/`cache`/`shim`/`unshim` are nub-own management verbs — same status as `nub node …` and `nub upgrade` above. pnpm's `pm` is a *builtin-forcing prefix* (`pnpm pm <x>` runs the built-in `<x>`, bypassing a same-named script) with NO subcommands of its own, so there is nothing under it to mirror and no pnpm-compat obligation binds these verbs. Don't let a compat audit flag `pin`/`migrate` as pnpm-grammar violations.
- **Lockfile-compatibility is a SEPARATE axis from CLI-compat.** nub is lockfile-compatible with whatever the project already uses (npm/pnpm/bun round-trip, yarn READ-ONLY) — it does not impose its own lockfile format. "pnpm-only" governs the CLI grammar; the lockfile axis is multi-PM.
- **Reversed 2026-06-27 — yarn berry write is now in scope** (maintainer's call; see [`.fray/pnp-node-linker.md`](.fray/pnp-node-linker.md) D1). Nub is building berry `yarn.lock` + `.yarn/cache/*.zip` + `.pnp.cjs` write for the Yarn Plug'n'Play node-linker, so the once-final read-only stance below no longer holds for berry. Most of the "structurally hard" rationale dissolved under empirical scrutiny: virtuals aren't serialized into the lock, `compressionLevel` defaults to STORE so there's no deflate-matching, and `.pnp.cjs` is ungated by `--immutable`. The effort accepts lockfile churn — operability, not byte-lossless `--immutable`-clean parity — which is what makes it tractable (see `wiki/research/pnp-write-fidelity.md` and thread decisions D2/D8/D9). A clean rewrite of this bullet is step 0 of the PnP implementation epic; it's annotated rather than rewritten here pending that pass. yarn-classic (v1) write stays out, and the pnpm-only CLI-grammar axis is unaffected. Superseded rule, retained for context: nub previously refused every `yarn.lock` write — the mutating-install family (`install`/`add`/`remove`/`update`/`dedupe`/`ci`) was gated to reject any operation that would rewrite it, and `nub pm use yarn` (one-shot identity switch) writes a classic-v1 yarn.lock on conversion. Why it was thought settled: berry's `yarn.lock` is resolution-protocol-coupled (`__metadata` cacheKey, per-entry `languageName`/`linkType`/`checksum`, berry-format `resolution:` descriptors, PnP/`.pnp.cjs` coupling), so a lossless round-trip would mean reproducing berry's checksum, linkType, and descriptor normalization exactly or `yarn install --immutable` rejects it — the PnP effort sidesteps that by accepting churn instead.
- **The PM is the vendored `aube` engine, embedded as a Rust LIBRARY** (path dep `vendor/aube`, called in-process — no aube subprocess). As of 2026-06-22, `vendor/aube` is **plain tracked files in nub's history** (Pattern B), NOT a submodule — an aube edit is a normal nub commit/PR; there is no pin to bump. Upstream = `jdx/aube`; the `nubjs/aube` `nub-fork` branch is retained as the upstream-PR staging area + historical record. **NEVER upstream anything to `jdx/aube` — and never even OFFER, propose, or label something an "upstream candidate" — without the maintainer's EXPLICIT, in-the-moment instruction (HARD RULE, decided 2026-06-28).** Default: do NOT upstream and do NOT mention upstreaming anywhere (not in PR bodies, commit messages, or reports). The contribution mechanics below exist only for when the maintainer explicitly asks; absent that, they stay dormant. Fork changes are default-preserving (standalone aube byte-for-byte unchanged; the nub embedder opts in) or a justified latent-bug fix.
- **Config identity model:** a **nub-identity** project consumes ONLY neutral, cross-tool config (`overrides`, `resolutions`, `catalog`, `workspaces`) — never another PM's branded fields/files. A **compat** project (incumbent npm/pnpm/yarn/bun) is MIRRORED faithfully, branded fields included. pnpm-NAMED files are never read unless pnpm is the incumbent.
- **Project `nub.jsonc` is an intentional config surface and an explicit naming exception.** The project-root file is Nub's canonical typed runtime/install/dlx configuration, separate from global `~/.config/nub/nub.jsonc`. Its parser and disabled discovery gate land before the user-facing release; do not remove the reader as dead code or unflag discovery incidentally. Sandbox values may be parsed while remaining inert until their dedicated frontends.
- **Compat targets are PER-MAJOR-VERSION of each package manager (decided 2026-06-20).** Each major version is its OWN compat target because config homes, lockfile formats, and behavior shift across majors — the canonical example: **pnpm 10 reads/writes scalar settings from `.npmrc`; pnpm 11 moved them to `pnpm-workspace.yaml`/global `config.yaml`.** So treat **npm**, **pnpm 9**, **pnpm 10**, **pnpm 11**, **yarn 1 (classic)**, **yarn 2/3/4 (berry — lump ONLY if byte-identical behavior)**, and **bun** as distinct targets. When MIRRORING an incumbent (config-writes, lockfile, behavior), nub DETECTS the incumbent's specific major version (`packageManager` field → `devEngines` → installed-PM `--version` → lockfile-version signal) and mirrors THAT major — never assumes "latest." When the version is unknown but the PM is known, default to the **dominant / most-compatible** major (e.g. pnpm → the v10 `.npmrc` model, the larger install base + the lowest-common-denominator that even v11 still reads for auth). Architect this as a version→model MAP, not a two-branch special-case, so a new major slots in cleanly. (This governs the multi-PM compat-mirroring axis; the CLI-grammar axis stays pnpm-only and tracks pnpm's current stable.) **WITHIN REASON:** split a major into its own target ONLY where its behavior MATERIALLY differs AND that version is in real-world use; LUMP majors that are identical/near-identical; do NOT engineer distinct targets for versions nobody runs or speculative future majors. The map is the architecture so new targets slot in cheaply — not a mandate to pre-populate every historical/hypothetical major. Cover what people actually use (today: pnpm 9/10/11, npm, yarn-classic, yarn-berry, bun), pragmatically.
- **Brand boundary (public surfaces only):** no public `globalThis.nub`, no `nub:*` module namespace, no `@nub/*` scope, no `"nub"` user-authored config field, no documented `NUB_*` user knob. Internal mechanism is EXEMPT — internal `NUB_*`/`__NUB_*` plumbing vars, error codes, cache/dir names may carry the brand.
- **`node` PATH-hijack contract (under active design — see [[node-hijack-compat-design]]):** when something invokes `node` via nub's shim it should RESPECT node/PATH semantics (the hijack's legitimate job is version resolution/pinning); augmentation belongs to the `nub`/`nubx` entrypoints. `--node` is the explicit zero-augmentation escape hatch on `run`/`nubx`.
- **v0 verb surface:** `nub <file>` · `run` · `watch` · `nubx` / `nub dlx` / `nub x` · `upgrade` · `init` (nub-own TS-first scaffold, exception 4 above) · `nub node` (version management) · the pnpm-compatible PM (`install`/`i`/`ci` + `add`/`remove`/`update`/`import`/`dedupe`/`pm …`). No `nub node <file>` passthrough; `--node` flag is live, and a truthy `NODE_COMPAT` env var is the tree-wide augmentation opt-out (= `--node`, persistent).

## The repo is PUBLIC — never commit internal discussion (HIGH PRIORITY)

`github.com/nubjs/nub` is a **public open-source repo with real stars**. Everything committed — code, comments, commit messages, `tests/**` READMEs, every tracked markdown file — is world-readable forever, and **survives in git history even after you delete it**. So: **never put internal product / marketing / competitive discussion or considerations into a tracked file.** Banned from anything git tracks:

- **Benchmark-presentation strategy** — which competitors or numbers to show vs. omit, "include X only where we're competitive," competitive-band thresholds, anything about how results are framed for publication.
- **Candid hedges that read as gaming** — "load-bearing for honesty," "this is the headline," framing a default (e.g. GVS auto-disable for next/nuxt/parcel) as something to be sheepish about, or any sentence a competitor/skeptic could screenshot as "they juice their benchmarks."
- **Private attributions / quotes / first-person strategy** — by-name attributions, internal quotes, "our number," "don't publish X," "reads as cherry-picking." Public copy states what the code DOES, factually and neutrally — never *why a number was framed a certain way*.

**Commit messages are professional, factual, and neutral — NEVER braggy, competitive, or editorializing.** A commit message states WHAT CHANGED, not how great it is or how it stacks up against a competitor. Banned: competitive bragging ("nub clearly beats bun", "crushes pnpm", "fastest"), superlatives/hype, and any framing a skeptic could screenshot. Write `site: update warm-install bench to 36-run numbers`, NOT `site: … (nub clearly beats bun)`. Benchmark commits name the change (new run, new sample size, updated bars) — never the verdict. This applies to every commit subject + body; the rule is the same as for any tracked file (above), and commit messages are world-readable forever and cannot be cleanly scrubbed once pushed.

All internal deliberation lives in the **LOCAL-ONLY, gitignored** surfaces — `.fray/`, `epics/`, `wiki/` (none are published). Keep strategy, competitive analysis, and decision rationale there, never in a tracked source/doc/test file. When in doubt, leave it out of the commit.

## Git & GitHub maintainer hygiene (HIGH PRIORITY)

nubjs/nub is a public open-source project with real reporters and contributors watching. Maintainer responsiveness on issues and PRs is a visible signal of the project's health, so these are not optional courtesies — they are mandatory hygiene. The end-to-end issue playbook lives in the `address-issue` skill (`.claude/skills/address-issue/SKILL.md`); the GitHub-comment tone + the durable conventions below are also captured in the cross-project guide [`PROSE.md`](PROSE.md). Every comment here follows the project's existing tone bar — factual, neutral, professional, never braggy or competitive (see [The repo is PUBLIC](#the-repo-is-public--never-commit-internal-discussion-high-priority) and the commit-message tone rule); these rules do not relax it.

- **Initial ack of an EXTERNAL issue = EXACTLY the comment `Investigating.` — nothing more (HIGH PRIORITY).** When you begin work on an EXTERNAL (non-maintainer) issue, post one comment whose entire body is `Investigating.` so the reporter knows it's been seen. Do NOT pad it with a summary, a plan, a timeline, or a restatement of the bug. (Internal/self-filed issues don't need the acknowledgement.)
- **Substantive issue/PR comments are TERSE + FACTUAL per PROSE.md — never a verbose essay (HIGH PRIORITY).** A substantive comment states what you found and what you did, in the fewest words that carry the facts; no preamble, no editorializing, no walls of text. NEVER write "previous comments were wrong" or similar meta-commentary — especially since prior comments are often an automated bot's, not a human's. Posting AS the maintainer's account is fine; the comment just has to be reasonable. The `gh-comment-guard` PreToolUse hook (`.claude/hooks/gh-comment-guard.mjs`, wired in `.claude/settings.json`) enforces this — it BLOCKS an over-long `gh issue/pr comment` / `gh pr create` body; trim per PROSE.md, or set `NUB_ALLOW_LONG_COMMENT=1` only for a genuinely-needed longer body (e.g. a detailed release note).
- **ALWAYS, NON-NEGOTIABLE (#1 PR-hygiene rule): a PR that resolves an issue MUST carry a closing keyword (`Closes #123` / `Fixes #123` / `Resolves #123`) in its BODY before it is opened.** This is not ordinary hygiene — it is a hard, can't-miss requirement: the closing keyword is what makes the merge auto-close the issue, and a fix-PR that lands without it leaves a resolved issue silently open (a maintainer-visible failure). Verify the keyword is in the body as the LAST step before `gh pr create` — every time, no exceptions. A PR that merely RELATES to an issue (touches the area, partial work, follow-up) uses `Refs #123` instead (no auto-close). Never land a fix-PR that leaves its issue unlinked, and never open a resolving PR without the closing keyword.
- **On EVERY release, comment the release link + version on ALL closed issues and merged PRs that shipped in it.** This is MANDATORY maintainer hygiene — always do it, no matter what. For each issue closed and each PR merged since the previous release tag, post a comment with the version it shipped in and a link to the release, e.g. `Shipped in v0.1.8: https://github.com/nubjs/nub/releases/tag/v0.1.8`. The fix being merged is not the same as it being on a published binary; this comment closes that gap for the reporter. (The `release` skill's Step 5 carries the concrete enumeration + commands.)
- **Close issues with a brief factual comment — never silently.** Use `gh issue close <number> --comment <text>`, stating what fixed it or why no code fix is needed.
- **Commit-msg hook strips agent co-author trailers automatically.** `.githooks/commit-msg` removes `Co-authored-by` trailers naming automated agents (Claude, Anthropic, noreply@anthropic) before any commit lands. `core.hooksPath` is set to `.githooks` in `.git/config` — shared across all worktrees of this clone, zero per-worktree setup. A fresh clone needs one command: `git config core.hooksPath .githooks`. The CI `no-agent-trailers` gate (`.github/workflows/commit-hygiene.yml`) is the backstop for anything that slips through. Scope: UI-visible co-author avatar trailers only — `Claude-Session:` URLs and other body lines are NOT flagged by either the hook or the CI gate.

## What Nub is, briefly

- **A Rust CLI that orchestrates the user's installed Node.** v0 is a fast script runner plus a TypeScript-just-works runtime. Mechanism: Node's own extension surfaces — `module.registerHooks()`, `--import` preload, `--require`, env vars, N-API addons, V8 flag injection.
- **TypeScript-first developer supertool.** Own transpiler (oxc-based). TS, JSX, non-erasable syntax, `emitDecoratorMetadata` all supported.
- **Compatibility is paramount.** Code targeting Node must run on Nub byte-for-byte. The contract: **default mode = Nub runtime augmentation active; `NODE_COMPAT=1` or `--node` = augmentation disabled, plain Node behavior**.
- **Behavioral additivity** is the rule for augmentation — [`wiki/philosophy.md#additivity`](wiki/philosophy.md#additivity) for the canonical statement. "Additive" means doesn't break code written for vanilla Node; mechanism is restricted to Node's own extension surfaces (not source patches).

## The augmenter / fork distinction is load-bearing

A previous direction (2026-05-17) positioned Nub as a soft fork of Node, with patches to Node source on the table. **That direction was reversed 2026-05-18.** Nub is now a Rust CLI augmenter via Node's extension surfaces only. Source patches are off the table.

If you're editing a plan doc and find "soft fork" or "patched Node" language, it predates 2026-05-18 and should be updated. The authoritative statement of the current direction is [`wiki/architecture.md#augmenter-not-fork`](wiki/architecture.md#augmenter-not-fork).

The reason the distinction is load-bearing for agents specifically: implementation approaches that *sound* clean — "just patch `lib/internal/modules/esm/resolve.js` to do X" — were valid under the old posture and are no longer. The current mechanism rule is **"would a user on plain Node, plus the corresponding `module.register()` / `--import` / npm-addon, get the same result?"** If yes, in scope. If no, find a different mechanism or scope the feature out.

### The brand boundary — public surfaces only (internals are exempt)

The brand boundary protects **public-facing surfaces only** — anything a user imports, installs, types, or authors as configuration, or any API they call by a documented name. **Internal mechanisms are explicitly exempt:** error/warning codes, environment variables Nub sets for its own cross-process / shim plumbing, internal `globalThis.__nub*` sentinels, and internal cache/dir names — the brand may leak there freely and chasing it out is wasted effort. (Error codes and the internal cross-process env vars Nub sets to change shim behavior are internal mechanism; the concern is only the brand leaking into public-facing APIs — don't over-police the boundary for internals.) The rules below govern the *public* surface and are absolute there — do not add a public nub-branded API even "behind a flag."

- **No PUBLIC `globalThis.nub`.** Nub never injects a *documented, user-callable* global named after itself; `Bun.*`-style conveniences, if they exist at all, go in importable modules. Internal markers are FINE — a `globalThis.__nub*` sentinel used for runtime/cross-process plumbing that user code never references is not a leak (reversed 2026-06-11; the old "no internal markers either" was too strict).
- **No `nub:*` module namespace.** Nub never registers synthetic specifiers users would `import` (`nub:test`, `nub:sqlite`, `nub:serve`). The `node:*` namespace is upstream Node's and closed; Nub does not squat a sibling. (Public surface — users type it.)
- **No `@nub/*` npm scope** for user-facing packages or APIs — the bare `@nub` scope is unused entirely. The CLI ships under the project's own `@nubjs` org: `@nubjs/nub` (the package users `npm install -g`) plus its `@nubjs/nub-<platform>` `optionalDependencies`, which carry the per-platform binary + N-API addon and are install-time plumbing never imported by user code (same shape as `@biomejs/biome` + `@biomejs/cli-*`, `@rollup/rollup-*`, `@esbuild/*`). Nub's ambient TypeScript declarations ship as `@nubjs/types` — our own org, a types-only devDep, not a user-imported API. (We do not own the npm package `nub`, so `@types/nub` on DefinitelyTyped is not a viable home.) `@nubjs` is deliberately distinct from the prohibited bare `@nub`. See [`wiki/philosophy.md#the-brand-boundary`](wiki/philosophy.md#the-brand-boundary) for the full rationale.
- **Internal `NUB_*` env vars are FINE; never a brand env var as a documented USER knob.** Reversed 2026-06-11: Nub MAY set `NUB_*` / `__NUB_*` environment variables for its own cross-process plumbing — telling a child shim how to behave, detection sentinels, debug toggles — these are internal mechanism, not a public API, and the old "no `NUB_*` ever, it shows up in `printenv`" rule was over-strict. What remains: don't *document* a `NUB_*` var as a knob users are meant to set, and for genuinely user-facing toggles prefer `NODE_*` (where Node doesn't claim the name, e.g. `NODE_COMPAT`) and neutral conventions (`NO_COLOR`, `FORCE_COLOR`, `PORT`, `NODE_ENV`).
- **The nub runtime respects ZERO `AUBE_*` env vars — the embedded engine's brand is never nub's config surface (nub is the package manager).** DISTINCT from the exemption above. That exemption covers internal vars nub *sets* for its own children (detection sentinels, shim signaling — a user never touches them). THIS rule covers vars the nub runtime *reads from the environment to decide behavior* — those are user-facing knobs, and when aube runs *as nub* the user sees nub while aube is an invisible implementation detail, so an `AUBE_`-branded knob is a brand leak. aube currently exposes ~30 such runtime vars (`AUBE_DISABLE_CLONEDIR` — already documented as the clonefile opt-out — `AUBE_CONCURRENCY`, `AUBE_CACHE_DIR`, `AUBE_NO_UPDATE_CHECK`, `AUBE_PRIMER_PICK_GATE`, the `AUBE_DISABLE_*` perf toggles, `AUBE_DIAG_*`, `AUBE_SELF_*`/`AUBE_VERSIONS_HOST`, …); under nub, none of these may be the surface. Surface preference (now that nub fully owns package management, `NUB_*` is legitimate, no longer a last resort): use a **neutral npmrc field** where one reads naturally (config any tool could honor — also reachable as `npm_config_<field>`), else a **`NUB_*`** env var / nub-namespaced config is FINE for nub's own PM knobs (e.g. `NUB_CACHE_DIR`, `NUB_PRIMER_TTL`). The hard line stays: **never `AUBE_*`** (the embedded engine's brand) for a nub-facing knob. Two disciplines on top: (a) **keep the exposed set as SMALL as possible** — every knob is a maintenance + doc + support surface; expose one only on real demand, default the rest internal or dropped; (b) **design each knob to GENERALIZE, not special-case** — a `NUB_PRIMER_TTL` *duration* (default unlimited), never a boolean `primer_evergreen` flag (config should be as generalizable as possible; a binary is a degenerate special-case of the right abstraction). MECHANISM (structural fix, not var-by-var whack-a-mole): extend the **Embedder profile** — the same compile-time pluggability jdx merged in #862 — with an env-prefix / env-resolution hook, so standalone aube keeps reading `AUBE_*` while the NUB profile reads neutral/`npm_config_*`/`NUB_*`; the boundary is enforced once, structurally. (Build-time `AUBE_*` knobs in aube's `build.rs` — `AUBE_PRIMER_TOP`, `AUBE_REQUIRE_PRIMER`, `AUBE_PRIMER_GENERATED_AT` — are aube's own build machinery our CI sets; out of scope for this *runtime* rule, a low-priority consistency cleanup at most.) Audit + migration tracked as the `brand-boundary-env` thread.
- **Prefer neutral names for USER-AUTHORED config (a public surface), with one explicit file exception.** A `package.json` field a user writes is public, so Nub reads a neutral `"tasks"` field rather than a `"nub"` field; do not add a `"nub"` package.json namespace or `nub.toml`. The deliberate exception is project-root **`nub.jsonc`**, Nub's canonical typed runtime/install/dlx config file. Its name and staged discovery/unflag plan are settled; do not reinterpret this preference as a reason to remove or rename it. Nub may also adopt a vendor-neutral, any-tool-could-implement-it field by decision record — currently `"tasks"` ([`wiki/commands/tasks.md`](wiki/commands/tasks.md)).
- **Sandbox config capabilities are per source scope, not repository-trust guesses.** Project `nub.jsonc` and root-authored `scriptsMeta` may use approved dynamic env values and credential-broker injection; dependency-controlled `dependenciesMeta` may not. Filesystem substitution is unconditional. A mixed project/script/dependency policy must preserve those per-scope capabilities rather than applying one global trusted/untrusted boolean, and Nub never tries to infer whether a checkout or PR is trustworthy.
- **The brand boundary is SYMMETRIC — a true nub project consumes only un-branded, cross-tool config.** Just as Nub never *emits* its own brand into your config (no `"nub"` field), **when Nub is the project's active PM it never *consumes* config branded with another package manager** — `pnpm.overrides`, `pnpm.*`, yarn-namespaced config, Bun-branded fields, etc. are all ignored. A nub-identity project respects only the neutral fields any tool could read: `overrides`, `resolutions`, `catalog`, `workspaces`. (When the inferred active PM is npm/pnpm/yarn/bun — a *compat* project, not a nub project — Nub MIRRORS that PM faithfully, *including* its branded fields, because the project genuinely is that PM's. The no-other-brand rule is specifically for nub-identity projects.) This is the config-side of the brand boundary; the mechanism lives in the config-scoping policy (`crates/nub-cli/src/pm_engine/config_scope.rs`, "mirror the active PM, never silently").
- **pnpm-NAMED files and paths are NEVER read unless pnpm is the incumbent PM (non-negotiable).** When the project's incumbent package manager is NOT pnpm, Nub does not read any file or path that contains the literal string "pnpm" in its name — this includes `pnpm-workspace.yaml`, `.pnpmfile.cjs`, `.pnpmfile.mjs`, `.pnpmrc`, `~/.config/pnpm/` (including `~/.config/pnpm/auth.ini`), and any `pnpm.*` field in `package.json`. The gate is on the **NAME**, not the semantic effect — "it's auth-only / harmless / no resolution impact" exempts nothing. Crucial distinction: **pnpm-NAMED ≠ pnpm-specific.** A *generically-named* field that only pnpm happens to support (e.g. `overrides`) is a separate, case-by-case decision — it is not pnpm-branded by name and may be honored in a nub-identity project. The file-name half of the symmetric boundary: just as Nub never emits pnpm-branded names into your config, it never reads them unless pnpm is provably the active PM.
- **Error codes are internal — don't brand-sweep them.** `ERR_NUB_*` / `ERR_AUBE_*` / `WARN_*` identifiers (and the engine errors that surface them) are internal mechanism; a brand string in a code or message is acceptable and not worth chasing. *Adding* a missing code is good UX; rebranding an existing one for purity is not required.
- **No vendored Node patches.** Nub does not patch Node source, ship a custom-built Node binary, or embed `libnode`. Every augmentation rides on Node's public extension surfaces. (This is an architecture/additivity rule, not a brand rule.) If a feature can't be expressed through those, find a different feature.

### Web/runtime API compliance bar — spec and Cloudflare semantics; Bun parity is not the goal

**Web/runtime APIs are implemented to the SPEC, or to Cloudflare's semantics — never to match Bun's non-spec additions.** When nub provides a web-platform or runtime API (Worker, HTMLRewriter, fetch, streams, etc.), the bar is WHATWG/TC39/WinterTC spec compliance, or Cloudflare Workers semantics where there is no formal spec (the documented HTMLRewriter exception). Bun-specific behaviors that are NOT spec-compliant are NOT implemented for parity's sake — do not add an API or a behavior solely because Bun has it. If a user later asks for a specific Bun behavior, it's considered then, individually, on its merits. Spec/Cloudflare compliance is the goal; Bun parity is not. (Bun is still useful as a reference for what's POSSIBLE and as a differential test oracle — just not as a spec.)

### `nub`-named npm packages, explained

Two families of `nub`-named packages exist on npm. Neither is the prohibited bare `@nub/*` scope, and neither is user-facing application code:

1. **`@nubjs/types`** — Nub's ambient TypeScript declarations ship as a types-only package under the `@nubjs` org we own and control. (We do NOT own the npm package `nub`, so `@types/nub` on DefinitelyTyped is not a viable home — `@types/<pkg>` maps to an npm package `<pkg>` we don't control, and DefinitelyTyped is community-controlled. The `@types/*` auto-load is also not the right pickup mechanism.) `@nubjs/types` is the correct home for any declarations user code references — `import.meta.hot` for hot-mode consumers, Nub-specific globals, etc. — as a devDep, never a runtime import. Brand-clean: under our own org, no bare `@nub` claim.

2. **`@nubjs/nub` + `@nubjs/nub-<platform>`** — the CLI itself and its platform-specific binary distribution packages (`@nubjs/nub-darwin-arm64`, `@nubjs/nub-linux-x64`, … — 8 platforms total). `@nubjs/nub` is the package users `npm install -g`; the `@nubjs/nub-<platform>` packages are its `optionalDependencies`, each containing the compiled Rust binary + N-API addon for one platform, selected at install time by npm's `os`/`cpu` filters and copied into place by `postinstall.js`. They are install-time plumbing — never `import`ed, never referenced in application source, never a thing users type. Same distribution pattern as `@biomejs/biome` + `@biomejs/cli-*`, `@oxlint/cli-*`, `@rollup/rollup-*`, `@esbuild/*`. The org scope is `@nubjs` (the project's own npm org), deliberately distinct from the prohibited bare `@nub`.

The earlier dismissal of `NODE_COMPAT=1` was about a *file-run compat* var (superseded by the `--node` flag); this revival is a DIFFERENT thing — a tree-wide augmentation kill-switch, brand-clean (`NODE_` prefix, Node doesn't claim the name). Full decision record: [`wiki/commands/node.md`](wiki/commands/node.md).

## The aube vendoring + upstreaming — MEMORIZE THIS (forgotten twice; ground every aube claim here, not memory)

nub's package manager IS the vendored **aube** engine, embedded as a Rust LIBRARY. The structure (re-read this before asserting anything about how the integration works):

- **Embed, not subprocess.** `vendor/aube` is a path dependency — **plain in-tree vendored files** (NOT a submodule anymore; the gitlink was dropped 2026-06-22, Pattern B). nub has its OWN CLI (clap dispatch + `pm_engine::ENGINE_VERBS`) and calls `aube::commands::<verb>::run(typed_opts)` IN-PROCESS. There is NO aube subprocess and NO aube CLI under nub; aube's own `cli_main` + tool subcommands are DEAD under nub. All engine output is rebranded through `crates/nub-cli/src/pm_engine/present.rs` (`ERR_AUBE_*`→`ERR_NUB_*`, `aube`→`nub`, jdx URLs stripped).
- **Fork vs upstream.** The vendored source originates from the fork **`nubjs/aube`**; upstream is **`jdx/aube`**. The vendored tree lives directly in nub's history under `vendor/aube/` — editing it is a normal nub commit; there is no pin to bump. The `nubjs/aube` `nub-fork` branch is retained as the upstream-PR staging area + historical record, NOT the source of truth (nub `main` is).
- **UPSTREAMING — contribution PRs target `jdx/aube` (UPSTREAM), NEVER the fork (the thing forgotten 2026-06-14).** A contribution PR is `jdx/aube:main ← nubjs/aube:<branch>` (e.g. #888 = `embedder-profile-fixes`; #897 = the `process-exit-sweep` stack). Do NOT open a fork-internal `nubjs/aube ← nubjs/aube` PR to upstream — it upstreams nothing. **Stacked PRs also target `jdx/aube`** (base `main`; note the stack + which commit to review in the body; the diff auto-cleans once the parent PR merges). Cross-fork stack creation: `gh pr create --repo jdx/aube --base main --head nubjs:<branch>`.
- **What upstreams vs stays fork-only:** PLUGGABILITY / additive changes that are NO-OP for standalone aube (the embedder profile, env-resolution hooks, the `prog()`/`cmd()` source-branding helpers, the exit-code sweep) → upstream to jdx/aube. MATERIAL PM-behavior / nub-specific changes (primer-TTL, etc.) → stay in the fork (merge-synced on main, never rebased — see the workflow below), upstreamed later only if jdx wants them.
- **The embedder profile** (jdx merged it as #862): compile-time pluggability. nub sets its own profile (`cache_namespace` "nub/pm", `env_prefix`/`config_env_prefix`, banner-gating, pnpm-surface off, `read_branded_pnpm_config` gating). The NUB profile is what makes the brand + config boundary hold; standalone aube keeps its `AUBE_*` behavior.
- **Fork-discipline for EVERY vendor/aube change:** default-preserving (standalone aube byte-for-byte unchanged on the default path; the embedder opts in) OR a justified latent-bug-fix jdx would accept unconditionally. A trustworthy-build/test deliverable gets its OWN isolated clone + `CARGO_TARGET_DIR`. Because aube is now plain in-tree files, the change lands as a normal nub PR (the aube diff is visible + reviewable in the PR) — no separate pin-bump assembly step.

### The plain-vendoring workflow — aube edits are normal nub commits; upstream via split/format-patch (MEMORIZE)

As of 2026-06-22 (Pattern B), `vendor/aube` is plain tracked files in nub's own history — NOT a submodule. This removes the pin, the push-before-pin order footgun, and the submodule-init step from every flow. The three recipes (all PROVEN before the migration):

- **(1) Make an aube change = a normal nub edit/PR.** Edit `vendor/aube/...`, commit, open a PR to nub `main` like any other file. The aube diff shows in the PR (more reviewable than a pin bump ever was). No `nub-fork` branch, no pin, no push-before-pin order. Fork-discipline (above) still applies: default-preserving or a justified latent-bug-fix.
- **(2) Upstream selected aube changes to `jdx/aube`.** Two equivalent proven methods; prefer (2b) for a few cherry-picked commits, (2a) for a larger contiguous slice. The contribution PR still targets `jdx/aube:main` from a branch off it (NEVER a fork-internal PR), so the upstream diff is only the change — never the whole fork delta.
  - **(2a) `subtree split` → cherry-pick onto a fresh branch off upstream:**
    ```sh
    git subtree split --prefix=vendor/aube -b _aube_split   # re-roots paths to aube's own layout, drops nub-only commits
    git fetch upstream main                                  # upstream = jdx/aube
    git checkout -b upstream-<topic> upstream/main
    git cherry-pick <sha-from-_aube_split>...               # a MIXED commit brings ONLY its aube hunk across
    git push fork upstream-<topic>
    gh pr create --repo jdx/aube --base main --head nubjs:upstream-<topic>
    ```
  - **(2b) `format-patch --relative` (no split branch needed for a few commits):**
    ```sh
    git format-patch <sha1>..<sha2> --relative=vendor/aube -o /tmp/aube-patches   # re-roots every diff path
    git checkout -b upstream-<topic> upstream/main
    git am --3way /tmp/aube-patches/*.patch                                       # resolve conflicts normally
    ```
- **(3) Sync FROM upstream `jdx/aube`** — infrequent + deliberate. Re-vendor the upstream tree and commit the delta as ONE reviewable nub commit; the upstream delta is visible in the PR. Keep the fork delta THIN by upstreaming aggressively (recipe 2) so this merge stays cheap:
  ```sh
  git -C <clone-of-jdx-aube> archive main | tar -x -C vendor/aube
  git add vendor/aube && git commit -m "aube: sync upstream <sha>"
  # reconcile any nub-specific fork delta on top (cherry-pick / am from the prior vendored state)
  ```
- **`nub-fork` is the upstream-PR staging area, NOT the source of truth.** nub `main` holds the canonical vendored aube. `nub-fork` on `nubjs/aube` is kept as the place upstream-PR branches are pushed FROM (recipe 2) + the historical record — it is no longer where day-to-day fork work lands.

## Testing philosophy

**Minimum number of tests, comprehensively covering the API surface.** This is the load-bearing rule for the v0.1 implementation, and the antidote to AI-generated test bloat. Quality of coverage matters; volume of tests does not.

> **Before you push, run the pre-push local verification loop** (see [Implementation quality discipline](#implementation-quality-discipline) below). Start with `make verify` for the bounded host-local gate: root and native formatting, all-target/all-feature clippy, the native addon, the static brand lint, and the ordinary test suite. It does not reproduce platform matrices, Docker jobs, or change-specific end-to-end tests; run those separately when the change requires them, then promote durable checks into the suite. Get it green locally and push ONCE — pushing fix-after-fix to "see if CI passes" starves the shared runner pool.

- **Comprehensive coverage, not exhaustive coverage.** Each behavior gets tested once, well, against the cases that matter: golden path, the one or two failure modes that have user-facing implications, the boundary condition that's easy to get wrong. Stop there. Don't write `it("should not throw when input is null")` after you've already written `it("rejects null input")`.
- **Hand-crafted feel.** Tests must read as if a careful engineer wrote them, not as if an agent enumerated possibilities. Concrete signs of agent bloat to avoid: identical assertions across multiple `describe` blocks with different setup; per-input parametrization where one assertion would do; test names that paraphrase the implementation instead of describing the contract; descriptions like "should handle X" without naming what "handle" means.
- **Carefully considered abstractions.** Test helpers and fixtures are part of the contract. Don't bury behavior in clever shared setup; don't copy-paste either. Pick the shape that lets a reader skim a test file in 30 seconds and understand what's being verified. If a test file is over 300 lines, you're probably testing too many things in one place or each test is too verbose — split or trim.
- **Some things are untestable, and that's fine.** Perf-shaped behavior, OS-specific corners, race conditions that require infrastructure we don't have, behaviors that depend on Node's internal scheduling. Leaving them untested is honest; writing ceremonial fake tests for them is sloppification. Document the gap in a comment if it'll surprise a future reader.
- **Pull from upstream where it makes sense.** Node ships an extensive test suite at `nodejs/node/test/`. The executable-level subset (tests that just spawn `node` and assert behavior) is the strongest possible compat-validation against Nub. We run it black-box in two modes: `nub --node` (expected ~100% pass — passthrough mode) and `nub` augmented (expected near-100% with documented divergences). See `wiki/research/node-test-suite-leverage.md` for the harness design and CI cadence.
- **Test names describe the contract, not the implementation.** `test("emits ECONNRESET when peer closes mid-write")` is good; `test("emits ECONNRESET")` is too short to mean anything; `test("calls socket.destroy() with the EconnReset constructor when..."` is paraphrasing the code.
- **Failure messages must be self-debugging.** When a test fails in CI, the assertion message should make the cause obvious without rerunning locally. Asserting `expect(result).toEqual({ ok: true })` is fine when it passes and useless when it fails ("expected {ok: true}, got {ok: false}"). Use `expect(result.ok).toBe(true)` or write a custom message.
- **Flakes are hunted and killed at the SOURCE — never papered over (HIGH PRIORITY).** A test that passes on one run and fails on another (green on main, red on a PR; passes locally, fails on Windows CI) is a FLAKE, and a flake is a P1 defect in the test, not noise to retry past. The fix is ALWAYS to find and remove the nondeterminism at its root — almost always **shared mutable state or test-ordering coupling**: a test that mutates a process global (`std::env`, `current_dir`, a `static`/`OnceCell`/`lazy_static`), shares a temp fixture/cache dir with a sibling, or relies on filesystem/iteration ordering, so parallel/interleaved execution under cargo's default multi-thread runner leaks state between tests. Make the test HERMETIC (isolate its env/cwd/fixture per-test; serialize only as a last resort) so it passes under any ordering and any `--test-threads`. BANNED as a "fix": `#[ignore]`-ing it, adding a retry/re-run wrapper, loosening the assertion to mask the divergence, or treating "re-run until green" as resolution (a CI re-run is only ever to UNBLOCK an unrelated PR held by someone else's flake — never the flake's fix). A single green run does NOT prove a flake fixed — re-run the test repeatedly (vary `--test-threads`, randomize order; for a platform-only flake use the `ci-adhoc-test` branch-scoped runner across multiple runs) before declaring it dead. Platform-specific surfacing (Windows-only) is a timing/ordering TELL, not evidence of a platform parse bug — chase the shared state, not the platform.

## Docker is available — use it instead of declaring things "untestable"

**Docker may be available.** Before you write off a behavior as "hard to test" or "can't verify locally," check whether a container closes the gap — several things the testing philosophy lists as untestable corners actually *are* reachable through Docker:

- **A clean, dependency-free environment.** A fresh `node:22.15-slim` (or no-Node `debian:slim`) container with no `~/.cache/nub`, no global Node, no local dev state — the honest way to test first-run install, the curl `install.sh` flow, `nub upgrade`'s `~/.nub` channel, provisioning a Node from `nodejs.org`, and "works on a clean machine."
- **A specific / floor Node version.** Pin `node:22.15` to exercise the **floor-only** defects a modern host Node (24/26) masks — `using`/`await using` down-leveling, the Temporal `toTemporalInstant` path, version-gated polyfills, the async-tier `module.register` loader. This is the local complement to the Node-22.15 CI leg.
- **Linux-specific paths.** musl-vs-glibc detection (`node:22-alpine` vs `node:22-slim`), arch matching, signal/`PATH` behavior on Linux.

**What Docker does NOT give you:** it runs **Linux containers only** (on a macOS/arm64 host → `linux/aarch64` by default, `--platform linux/amd64` via slow QEMU emulation). **Windows containers require a Windows host**, so cmd.exe behavior, `--script-shell` shell selection, `.cmd`/`.bat` resolution, and `nub.exe` vs `bin/nub` still can only be verified on the **`windows-latest` CI leg** — Docker is not a substitute there. When an item's *Verify* says "on Windows," it means Windows CI; do not claim a Docker run verified it. For a macOS- or Windows-only behavior you want to probe ad-hoc, use the **`ci-adhoc-test` skill** (`.claude/skills/ci-adhoc-test/SKILL.md`): a self-contained probe harness under `tests/<probe>/` driven by a **branch-scoped** workflow (`workflow_dispatch` + `push` to the branch) — **no PR required**, and not gated on PR state.

Keep containers ephemeral (`docker run --rm`), mount the repo read-only where you only need to read it, and never leave long-running containers behind.

## Performance tracing the package manager — the instrumentation already exists; turn it on (HIGH PRIORITY — load this BEFORE reverse-engineering a slow install)

When a `nub install` / PM operation is "mysteriously slow," do NOT trace Rust by hand — the **`pm-perf-tracing` skill** (`.claude/skills/pm-perf-tracing/SKILL.md`) is the canonical method, and it is what root-caused the hoisted-linker slowness (10.8s link → a single-threaded per-file `std::fs::copy` loop with the whole-dir `clonefile` fast path never firing). The essentials: `RUST_LOG=debug nub install` emits `phase:<name> <elapsed>` lines (the coarse resolve/fetch/**link** split) and works under nub today; the per-file/per-strategy linker diagnostic (`aube_util::diag` → `AUBE_DIAG_FILE`, the `link_clonedir`/`link_macos_small_copy`/… tally) EXISTS in aube but is gated off under nub (`env_prefix: None` in `pm_engine/identity.rs`) — flip it to `Some("NUB")` in a throwaway build to activate it. Judge by the **strategy tally + an A/B ratio** (e.g. default-hoisted vs `--node-linker isolated`), not the contended-host absolute, and always run on a **verified-clean warm `--offline` loop with rc=0**. (The `dev-tracing-telemetry` thread is making this a first-class, release-stripped toggle with chrome-trace/flamegraph export — prefer it once landed.)

## Probing methodology — differential fixtures, empirical over source (learned 2026-06-10)

The highest-yield way to find correctness bugs in nub is a **differential fixture**: a minimal fixture isolating ONE behavior, run against nub AND the reference tools it claims parity with (npm / pnpm / bun / node) on identical inputs — a divergence is the finding. This consistently beats happy-path app rounds: days of running real apps under nub surfaced almost nothing, while one five-minute `.env`-visibility fixture found two real bugs (a secret-leak over-reach where nub injected `.env` into non-node tools, and a NODE_ENV-cascade bug). Build the fixture, run every relevant tool, diff. Always compare against the thing you assert parity with — "nub does X" is unverified until "…and npm/pnpm/bun do Y on the same fixture" is in hand.

Three supporting disciplines, each earned the hard way:

- **Test empirically before reading source, and before deciding.** A throwaway fixture in a sandbox answers "what does it actually do?" faster and more reliably than tracing Rust/Zig. When a decision hinges on a tool's real behavior, reproduce it — don't infer it from code.
- **Read the COMMENTS and the resolution, not just the issue body.** A bug's body says what someone wanted; the maintainer comments and *why it was closed* say what's true. An "open feature request" can be a deliberate rejection with the real rationale in the thread — Bun's auto-`.env`-for-scripts looked like unmet demand in the body, but the comments showed it was intentionally rejected, with the core maintainer's root-cause analysis. Bodies alone will make you confidently wrong.
- **Ground every claim about the system in code or an experiment — never memory.** A confidently-designed probe this session referenced `nub test`, which is not a command. Verify the actual surface (read `cli.rs`, or run it) before you build against it.

And: **reversing a decision on new evidence is correct, not flip-flopping.** Architecture calls routinely dissolve or flip once the facts arrive — a "minimal vs elegant" split collapsed when investigation showed the elegant option was also the only *correct* one; a "keep this feature / lean in" call reversed twice as comments and a fixture came in. Present the decision, then keep probing; let evidence move you rather than defending the first answer.

## Iterating across Node versions and tiers

nub's behavior splits by **tier** — the fast tier (Node 22.15+, sync `module.registerHooks`) and the compat tier (18.19–22.14, async loader-worker via `module.register`) take different code paths and break differently — so a green run on a single modern Node (often 26) routinely masks compat-tier and floor-only defects. nub discovers its Node from `PATH`, so the cheapest way to drive it onto a specific version is `PATH="$HOME/.nvm/versions/node/v20.19.0/bin:$PATH" nub …`; sweep several to cover both tiers, and reach for Docker (per the section above) for Linux + floor confirmation before trusting a result. **Do not claim cross-version support from one Node.**

**ALWAYS use the LATEST Node major (currently 26) as the recommended/example version in docs, examples, Dockerfiles, `@types/node`, and marketing — NEVER the tier floor (22.15) or an older line.** Every `FROM node:X`, `@types/node@X`, "install Node X", and version-pinned snippet shows the newest major. This is DISTINCT from version-FLOOR facts: the support floor (18.19) and the fast-tier classifier (22.15) are facts to STATE where relevant (the tier table, "requires Node X+" minimums) — they are facts, not the version to put in a `FROM`/install example. When a new Node major releases, the "latest" reference moves up. (Docs code samples use the `{{NODE_MAJOR}}`/`{{NODE_VERSION}}` tokens, which the `remark-node-version` plugin substitutes to the newest release at build time — prefer those over a hardcoded number where a code/inline-code sample needs the latest.)

Feature-specific harnesses that encode this loop live under `tests/<feature>/` — e.g. `tests/pnp/` (Yarn PnP) builds a real fixture (`make-fixture.sh`), runs a scenario matrix across Node versions on the host (`run-pnp-matrix.sh`) and on Linux in per-version containers (`docker-matrix.sh`), and documents the whole loop + the tier model in `tests/pnp/README.md`. When you build or extend a system for iterating on a hard-to-unit-test feature, document it in-place like that (a `README.md` next to the scripts, plus comments in the scripts themselves) so the next agent reproduces it in minutes instead of rediscovering it.

## Implementation quality discipline

**Quality over velocity. Always.** Don't move fast, check boxes, and ship stubs as complete implementations. The specific failure modes:

- **ALWAYS prefer the root-cause fix over a guardrail (HIGH PRIORITY — maintainer directive, 2026-07-21).** When a defect admits both a real fix (implement or repair the behavior correctly) and a cheaper mitigation (a warning, an error gate, a docs caveat, a bail-out), the real fix is the default and the recommendation — out of the gate, not as a "follow-up." Never rank a warn/error/document-around mitigation above the real fix because it is less work; the amount of work is not a factor in choosing the right fix. A mitigation is acceptable only when the real fix is genuinely infeasible (external blocker, needs an upstream decision) — and then it must be framed as a stopgap with the real fix as the primary tracked deliverable. (Origin: #484 — an investigation recommended "abort on pnpm `nodeLinker: hoisted` in workspaces now, implement real pnpm-parity hoisting later"; the correct recommendation was to implement the parity fix directly. This bullet coexists with "Do not overengineer" below: that rule is about not inventing unneeded scope, not about picking the cheap mitigation over the correct fix for a real defect.)
- **Do not overengineer.** Solve the demonstrated problem at its actual scale. Do not introduce generalized infrastructure, extra abstraction, speculative hardening, or adjacent cleanup without evidence that the added complexity earns its cost; when evaluating recommendations, "not worth doing" is a valid and preferred verdict.
- **Wait deliberately for sub-agent returns.** After dispatch, the root continues useful non-overlapping work. When returns are the remaining dependency and no useful root work remains, state the active count and one-line lane summaries, then call one blocking `wait_agent` with a substantive timeout. Do not end or yield assuming completion notifications will start a new root turn. Reconcile every result, then continue waiting as needed until all relevant agents are handled or new user input supersedes; never busy-poll with repeated short waits.
- **Never mark a task `[x]` without verifying the behavior end-to-end.** Running `cargo test` is necessary but not sufficient. If the task says "implement per-line stream prefixing matching pnpm," you must run `nub -r --stream run build` on a real fixture and compare the output to `pnpm -r --stream run build`. A green test suite with a stubbed implementation is worse than an unchecked task.
- **Never claim "parity" without evidence.** "Complete workspace parity with pnpm" means every flag, every edge case, every output format. If you haven't tested a flag, don't mark it done. If the implementation is a simplified version, say so explicitly in the task note — don't use language like "implemented" for something that's "scaffolded."
- **Name what you actually built.** If you shipped `Stdio::inherit()` with a header line, that's not "stream prefixing." If you partitioned into fixed batches, that's not "work-stealing." Use precise language so the next reader knows the actual state.
- **If you're completing 10 tasks per hour, you're not testing them.** A thoughtfully implemented and tested task takes 15-30 minutes. If a phase has 12 tasks and you completed it in 20 minutes, audit yourself.
- **Shipping a user-facing feature or behavior change MUST include updating the docs.** A feature is not "done" until `site/content/docs/` reflects it. The code change and the doc update are the same effort — do not close the task or mark it complete until both are landed.
- **VERIFY LOCALLY BEFORE PUSHING — don't outsource verification to CI (the pre-push loop) (HIGH PRIORITY — protects shared CI capacity).** CI is the backstop, not the first place a defect should surface. This is a RUNNER-CONTENTION rule, not just a quality one: the repo has a finite GitHub Actions runner pool, every push fires ~5 workflows × the 8-platform matrix (~30-40min each), and several open PRs pushing fix-after-fix to "see if CI is green" SATURATES the pool — head-commit jobs then queue for HOURS (this actually happened: a config PR's run sat queued ~1h40m with zero jobs running while fix-commits churned). A red CI on something locally-catchable (the #34 `clippy --all-targets` miss, a fmt diff) is doubly wasteful — it burns a full matrix AND a restart. So: get it green LOCALLY, push ONCE. Before pushing a code change, run the loop in your worktree:
  1. **Incremental build** in the worktree (own `CARGO_TARGET_DIR` so it's warm/fast across iterations) — `cargo build` for the touched crates.
  2. **Run EXACTLY what CI runs** for the cheap gates: `cargo clippy --all-targets --all-features -- -D warnings` (scoped `-p` WITHOUT `--all-targets` misses test-code lints — the exact #34 gap) AND `cargo fmt --check` AND the scoped `cargo test` for what changed. Match the invocations in `.github/workflows/ci.yml`, don't approximate them.
  3. **End-to-end test the SPECIFIC functionality with a tmp fixture** — build the release (or debug) `nub` binary and exercise the actual feature against a throwaway fixture in `/tmp`, diffing against the reference tool (npm/pnpm/yarn/bun/node) where parity is claimed. "Tests pass" is not "the feature works"; run the feature.
  4. **Use Docker for behavior that touches the GLOBAL cache / config / a clean machine** — global `~/.npmrc`/config homes, the CAS store, first-run/clean-env install, a Node-version floor — run it in an ephemeral `docker run --rm` container so host state doesn't mask or pollute the result (see the Docker section above). Linux-only; Windows still rides CI.
  5. **INCORPORATE the verification into the actual test suite where reasonable** — a tmp-fixture e2e check that caught a real behavior should become a committed integration test (or a documented harness under `tests/<feature>/`), not a one-off you run and discard. A throwaway probe verifies *this* change; a committed test prevents the *next* regression. Promote it when the behavior is stable and the fixture is reusable; leave it a throwaway only for genuinely one-shot/environback checks.
  Push only after the local loop is green. This is the implementer's half of the "scale review to blast radius" rule — local e2e + the incorporated test is what makes the reviewer's pass cheap and the CI run a formality.
- **Inline comments are SPARSE and DENSE — design, invariant, provenance only (HIGH PRIORITY — actively suppress the model's default verbosity).** The models we run — Opus especially — DEFAULT to over-commenting: line-by-line narration that restates in English what the code already says. That default is WRONG for this codebase and must be ACTIVELY cut, not indulged — bias hard toward FEWER comments, and treat the bar to ADD one as high. Every comment must earn its place by communicating one of: system DESIGN (how this piece fits the larger architecture), a non-obvious INVARIANT or constraint, or DECISION PROVENANCE (why it's done THIS way, what alternative was rejected, the gotcha that bit someone). Do NOT narrate what the code already says — no restating the function/variable name, no step-by-step play-by-play, no `// increment counter`, nothing a competent reader gets immediately from the code itself. A single dense comment at a subsystem or function boundary explaining the WHY beats scattered line-by-line narration. When in doubt, cut it — if the code needs a comment to be clear, improve the code first. This is the code-side of the prose-terseness rule (same discipline as the docs-terseness and chat-brevity rules above, applied to code comments).

## Chat responses: few words, expert tone

Chat replies should use the **minimum words that convey every point**. Tone: a brilliant subject-matter expert who is terse by nature. Sentence fragments, conversational shorthand, no preamble, no recap, no hedging. State conclusions; cut throat-clearing and restatements of the question. This governs chat replies only — docs/code keep their normal rigor. (Concision ≠ omission: keep every distinct point, just strip the words around it.) Same rule, with the full cross-surface tone bar, lives in [`PROSE.md`](PROSE.md).

## Chat responses: references must be CLICKABLE (HIGH PRIORITY)

The harness renders GitHub-flavored markdown in a terminal that supports OSC-8 hyperlinks, so EVERY reference you surface in chat is a markdown link, never a bare URL, a plain `#N`, or a backticked path. Be EAGER — every issue/PR, file, fray thread, design doc, or plan you mention gets a clickable link, every time.

- **EVERY occurrence — not just the first or the "new" one (MOST-VIOLATED — codified 2026-06-30 after a bare-`#N` slip).** Link EVERY issue/PR reference in the message: the fifth mention as much as the first, the ones inside a TABLE or LIST as much as prose, the ones you already linked in an EARLIER turn, and repeated references to the same number. The failure to kill: linking only the freshly-relevant PR (e.g. `#265`) while leaving the others (`#224`, `#264`) as bare `#N`. There is no "already-known so a bare number is fine" exception — if it's an issue/PR number on screen, it is a link. When several appear in one message, they ALL become links.
- **GitHub issues/PRs** — render `#N` as `[#N](https://github.com/<owner>/<repo>/issues/N)` (the `/issues/N` form redirects to `/pull/N`, so it works for both). Never emit a bare `#N`.
- **Local files / fray threads / docs / plans** — link via the reader's IDE URI scheme so a click opens the file in their editor: `[name](cursor://file/<absolute-path>)` or `[name](vscode://file/<absolute-path>)` (append `:<line>` to jump to a line). Use whichever scheme the reader's IDE uses; `cursor://` and `vscode://` are examples, not the only schemes.
- **Bare URLs** — wrap in `[text](url)`; never paste a raw URL where a link is meant.

(This governs chat output only. In tracked files, keep links relative/agent-agnostic and never hardcode a machine-specific absolute path or username.)

## User-facing copy: prose & tone

**The cross-project copywriting/prose/tone guide is [`PROSE.md`](PROSE.md) — read it before writing any GitHub issue/PR comment, docs page, blog/marketing copy, or release note.** It is also surfaced as the `prose-writing` skill (`.claude/skills/prose-writing/SKILL.md`), which auto-triggers on copy work and points back to PROSE.md as the canonical source (the skill never duplicates it). It is the single source of truth for register, sentence/heading mechanics (never start a sentence or heading with inline code, command-headed sections, etc.), scannability, inline-code-pileup avoidance, description-field rules, real-output-only mockups, GitHub-response tone, markdown mechanics (never hard-wrap paragraphs), and release-notes shape. The sections below carry ONLY the nub-SPECIFIC copy rules that layer on top of it.

**Any GENERAL copy-style feedback gets applied EVERYWHERE, not just at the one spot it was raised** — and gets recorded in `PROSE.md`, not here. When the feedback is a general writing rule (not a one-off fix to a single sentence), dispatch a sub-agent to sweep ALL docs (and the homepage/blog where it applies) and integrate the rule, then add it to the shared guide.

**User-facing AGENT instructions must be CODING-AGENT-AGNOSTIC — NEVER overfit to Claude / `.claude` (HIGH PRIORITY).** Any copy an arbitrary coding agent reads and *executes* — `start.md` (the adoption flow), the `nub agent skill` output, docs pages that tell an agent to do something — must accommodate ALL coding agents (Claude Code, Cursor, Codex/Copilot, Cline, …), not just Claude. Each manages its standing instructions differently (`.claude/skills/<name>/SKILL.md`, `.cursor/rules/`, `AGENTS.md`, `.github/copilot-instructions.md`, …). So NEVER hardcode a Claude-specific path/format as THE place to write something. Instead, instruct the running agent to follow ITS OWN conventions and the user's repo's existing skill/rules layout — the agent reading the instruction knows where its instructional context lives; lean on that. Claude-specific paths may appear ONLY as one *example among several* ("e.g. `.claude/skills/nub/SKILL.md` for Claude Code; your agent's equivalent otherwise"), never as the sole prescribed target. (This is distinct from the repo's OWN tooling — `.claude/skills/*` for THIS repo's agents is fine; the rule governs copy that ships to USERS' arbitrary agents.) Keep `nub agent skill` / `https://nubjs.com/skill.md` as the agent-neutral content source.

### nub docs specifics (`site/content/docs/`)

- **Register:** zod.dev — to the point, code-first, no marketing fluff inside docs pages.
- **Page slugs are command-aligned:** `/docs/run`, `/docs/pm`; the command-less file runner is `/docs/files`.
- **Features whose API surface relies on nub's OWN ambient TS declarations get a `<Callout>` at the very top:** tell the reader to install `@nubjs/types` as a devDep alongside `@types/node`, and to use `@types/node` **26**. Apply only to pages whose types actually come from `@nubjs/types`/`nub-env.d.ts` (data-format imports, `Worker`, `import.meta.hot`, …) — not pages typed entirely by `@types/node`; word it to match what that feature actually needs.
- **The `--node` escape hatch is introduced once, on its own surface** (the runtime overview's `--node` section, the node-command page) or as a documented flag of the command a page is about (`nub run --node`, `nubx --node`) — NEVER as a tangential "passing `--node` disables/skips this" note tacked onto a PnP / TypeScript / web-storage / etc. behavior section. (This is the nub instance of the shared "don't sprinkle a cross-cutting flag's asides across unrelated pages" rule.)
- **Know the rendered HTML/CSS before hand-rolling a code block in a React/MDX component.** This site's global CSS styles a bare `<code>` element as INLINE code (background, padding, rounded chip) — so a `<code>` nested inside a custom `<pre>` makes the whole block render like an inline-code pill, NOT a code block. A code block is a `<pre>` containing the text DIRECTLY — no inner `<code>` (`<pre>{`line1\nline2`}</pre>`). (Standard `<pre><code>` is semantically fine in the abstract, but under THIS site's CSS the `<code>` hijacks inline styling, so omit it for hand-built blocks — or use a real MDX code fence, which fumadocs styles correctly.) And never put `select-none` / per-span styling on parts of a code block — it makes those parts unselectable and reads as "weird, not a normal code block." A custom component's raw elements inherit the site's GLOBAL element styles; check how the thing actually renders (drive the browser / read the CSS) instead of assuming the markup is inert.

## Blog & marketing copy (`site/content/blog/`, homepage): nub specifics

The general structure rules (open with the thing working, code blocks carry the argument, no walls of text, no flowery phrasing, benchmarks as visuals, asides sparing, showcase protective refusals with real failure output) live in [`PROSE.md`](PROSE.md). The homepage is the canonical register — when a passage or code block already exists there, reuse it rather than rewriting it. The nub-specific layers:

- **Benchmarks use the homepage `<Bench>` component** (a registered global MDX component). The file-execution bench belongs to the file runner, not `nub run`.
- **Bailout commands are sub-section asides.** Surfaces that exist for completeness (`nub node`, `nub pm`) get a short `###` under the section whose implicit behavior they back up, never their own `##` — "a Node version manager you should never need" framing.
- **Frame replacements as toolchain, not "npm packages."** What nub displaces is project-level tooling — `nvm`, `corepack`, the PM CLI — much of which was never an npm package. Keep the PM's two compat axes distinct: pnpm-compatible CLI surface; lockfile-compatible with whatever the project already uses (npm / pnpm / bun round-trip, yarn read-only). Concrete product comparisons are encouraged ("the uv experience, for Node"; "uv for Node" is a good framing).
- **Marketing asides use the styled treatment** (`.blog-prose blockquote`); default blockquote styling is unacceptable on the marketing site.
- **Version-gated claims must trace to code, not the wiki.** Every site claim about a Node-version-gated feature must map to a named constant/function in `crates/nub-core/src/node/flags.rs` or `spawn.rs` (no symbol → the claim cannot ship). The wiki's `status: v0.1 "Confirmed"` tracks intent, not implementation — copy written from it shipped false claims (WebSocket, webstorage, 2026-06-10).
- **"Every supported Node version" is a banned phrase** unless the feature is `typeof`-feature-detected in the shared preload. Unflagged features are tier- and version-banded; state the floor per row instead of glossing it.
- **Brand-boundary copy.** "The brand stops at the binary boundary" is wiki-internal shorthand, never user-facing — and absolute "the name nub never appears anywhere" claims are false (`~/.local/share/nub/store`, `~/.cache/nub`, internal `NUB_*` plumbing vars, error codes). The public claim is about *public surfaces only*: no nub-branded API you import or call, no `nub:*` import namespace, no `@nub/*` scope, no `"nub"` config field you author. **Do NOT claim "no `NUB_*` environment variables" on the homepage or in any public-facing copy** — three sanctioned user-facing PM config knobs exist (`NUB_CACHE_DIR`, `NUB_CONCURRENCY`, `NUB_PRIMER_TTL`), publicly documented in the PM docs, so that claim is false. The correct public promise is narrower: no nub-specific *imported/callable* APIs, no `nub:*` namespace, no `@nub/*` scope, no `"nub"` config field. Internal `__NUB_*` plumbing vars and test seams (`NUB_PTY_SCENARIO`, `NUB_RELEASE_BASE_URL`, etc.) are not user knobs and do not affect the public claim. The only thing to actually avoid is the absolute "the name nub never appears anywhere." The promise is "zero nub-specific *public* APIs / no lock-in."
- **Protective-refusal demos use real nub output.** Capture from `target/release/nub` or ground the exact string in `crates/nub-cli/src/pm_engine/` — never invent. (The general pattern — show the refusal as a feature, red ❌ on failure lines, ✓ on success — is in the shared guide.)

## Releasing

Nub is published to npm as `@nubjs/nub` with 8 platform-specific binary packages. The release is fully automated via GitHub Actions.

```bash
make version V=0.0.6          # sets version in all 9 npm packages + Cargo.toml
make version-check             # verify consistency
git add -A && git commit -m "v0.0.6"
git tag v0.0.6
git push origin main --tags    # CI builds 8 platforms, publishes to npm, creates GitHub release
```

**Makefile targets:**
- `make version V=<ver>` — set version across all packages atomically.
- `make version-check` — verify all package versions are consistent.
- `make npm-build` — build + package for the current platform (local testing).
- `make npm-publish` — publish all packages manually (prefer CI).
- `make npm-publish-dry` — dry-run publish.

**CI release workflow** (`.github/workflows/release.yml`): triggers on `v*` tags. Builds for darwin-arm64, darwin-x64, linux-x64, linux-x64-musl, linux-arm64, linux-arm64-musl, win32-x64, win32-arm64. Publishes via npm OIDC trusted publishing (no secrets needed). Creates GitHub Release with binary artifacts.

**Version regime:** stay in `0.0.x` until ready for public launch. Bump to `0.1.0` only when the whitepaper, benchmarks, and install experience are polished.
